# AI Prompt Lab 修复清单与后续拓展路线

## 文档目的

本文件用于沉淀当前项目的系统性修复建议与后续功能扩展方向，方便后续按优先级推进。

适用范围：
- 后端：`backend/app`
- 前端：`frontend/src`
- 工程化与交付链路：构建、测试、迁移、日志、权限、安全

## 当前判断

项目已经具备较完整的产品骨架：
- Prompt 管理
- 单次执行与执行历史
- A/B 测试与批量测试
- 评论与版本评审
- 团队协作与共享
- AI 配置管理
- 统计面板
- 管理后台

但当前更适合先做一轮“安全性 + 核心流程一致性 + 工程化”收口，再继续堆叠新功能。否则后续功能越多，修复成本越高。

## 修复优先级总览

### P0：必须立即修复

1. JWT 密钥配置不安全
2. 评论链路存在越权与存储型 XSS 风险
3. AI 输出渲染存在 XSS 风险
4. 被禁用用户仍可继续访问受保护接口
5. AI 配置密钥处理链路混乱，测试连接不可靠
6. 用户级 API Key 明文存储并直接返回前端/管理端

### P1：应尽快修复

1. 模型选择与多模型对比实际未完全按用户选择生效
2. 团队共享 Prompt 的“查看/编辑/运行/测试”权限链路不一致
3. A/B 测试页面结果流不完整
4. Prompt 复制功能依赖列表接口未返回的字段
5. 多个列表接口存在低效总数统计

### P2：建议排期修复

1. 前端高价值敏感信息落在 `localStorage`
2. 新建 Prompt 草稿未按用户隔离
3. 自动加入团队链接缺少确认
4. 回复场景中的 `@提及` 逻辑异常
5. 构建、测试、迁移、日志体系不完整

## 详细修复清单

## P0 修复项

### 1. JWT 密钥配置不安全

- 严重级别：P0
- 位置：`backend/app/core/config.py:17`
- 关联代码：`backend/app/core/security.py:33-41`
- 问题说明：`SECRET_KEY` 默认值为空字符串，且启动时没有阻断。
- 风险：JWT 签发与校验的根信任不安全，理论上可伪造令牌。
- 修复建议：
- 启动时强制校验 `SECRET_KEY` 非空且达到最小强度。
- 在开发环境之外，缺失时直接启动失败。
- 补充 `.env` 示例说明，不允许回退到空值。

### 2. 评论接口存在越权访问风险

- 严重级别：P0
- 位置：`backend/app/api/comment.py:23-57,65-114,189-222`
- 问题说明：评论相关接口验证了 Prompt 是否存在，但没有统一校验当前用户是否有权访问该 Prompt。
- 风险：已登录用户若知道 `prompt_id`，可能对私有 Prompt 查看评论、发表评论或读取统计。
- 修复建议：
- 抽取与 `prompt.py` 一致的 Prompt 可访问性判断逻辑。
- 评论、统计、用户搜索等相关接口统一复用。
- 同时纳入团队共享权限判断。

### 3. 评论渲染存在存储型 XSS

- 严重级别：P0
- 位置：`frontend/src/components/PromptComments.vue:164,216,525-535`
- 问题说明：评论内容经过 `renderContent()` 简单替换后，直接用 `v-html` 渲染，没有做 HTML 转义或净化。
- 风险：恶意评论可在其他用户打开页面时执行脚本。
- 修复建议：
- 默认按纯文本渲染评论内容。
- 如果一定要支持富文本，必须引入严格的 HTML sanitize。
- `@提及` 建议改成文本切分渲染，而不是拼接 HTML 字符串。

### 4. AI 输出渲染存在 XSS

- 严重级别：P0
- 位置：`frontend/src/components/ResultViewer.vue:138-141,177-180`
- 关联代码：`frontend/src/utils/markdown.ts:37-41`
- 问题说明：AI 输出经 `marked()` 转换为 HTML 后，直接通过 `v-html` 注入页面。
- 风险：模型输出中的恶意 HTML/脚本可被执行。
- 修复建议：
- Markdown 渲染后增加 HTML sanitize。
- 明确禁止原始 HTML 透传，优先只允许安全白名单标签。
- 对“详情渲染”与“自动弹出渲染结果”链路统一处理。

### 5. 被禁用用户仍可访问接口

- 严重级别：P0
- 位置：`backend/app/core/deps.py:60-64`
- 问题说明：`get_current_active_user()` 没有检查 `is_active`，仅原样返回用户。
- 风险：管理员禁用用户后，旧 token 依然可继续使用。
- 修复建议：
- 在 `get_current_active_user()` 中显式校验 `current_user.is_active`。
- 对禁用用户返回 `403` 或 `401`，并在前端触发登出。

### 6. AI 配置密钥处理链路不可靠

- 严重级别：P0
- 位置：`backend/app/api/ai_config.py:37-50,73-81,132-138,149-159,214-224,255-257`
- 关联代码：`frontend/src/store/config.ts:103-112,164-215`
- 问题说明：
- 后端创建 AI 配置时会加密 `api_key`。
- 但读取时直接把加密后的内容回传前端。
- 编辑时前端再把这段密文回传。
- 测试连接接口又直接把密文拿去调用 AI 客户端。
- 风险：
- 测试连接会误报失败。
- 编辑流程不可信。
- 密钥显示与真实值语义混乱。
- 修复建议：
- 后端响应只返回脱敏后的 `api_key` 展示值，绝不返回密文或明文。
- 更新接口对“未改 key”和“改了 key”做明确分支。
- `/{config_id}/test` 必须先解密再调用。
- 前端表单需区分“展示占位值”和“用户新输入值”。

### 7. 用户级 API Key 明文存储与泄露

- 严重级别：P0
- 位置：`backend/app/models/user.py:14`
- 关联代码：`backend/app/api/auth.py:59-85,114-140,155-175,178-191,248-266`
- 关联代码：`backend/app/api/admin.py:49-69,92-110,150-168,225-243`
- 问题说明：用户模型中仍保留 `api_key` 字段，并通过登录、`/me`、管理员用户接口直接返回。
- 风险：
- 前端、日志链路、管理员接口都会接触用户第三方密钥。
- 管理员可以批量看到所有用户的密钥值。
- 修复建议：
- 明确废弃 `User.api_key` 或迁移为加密存储。
- 所有用户响应模型中移除 `api_key`。
- 如需兼容，至少先停止回传，再做数据迁移。

### 8. 加密服务存在危险回退默认值

- 严重级别：P0
- 位置：`backend/app/services/encryption_service.py:16-17`
- 问题说明：加密主密钥与盐存在硬编码回退值。
- 风险：环境变量未配置时，所有部署都会落到同一组已知密钥。
- 修复建议：
- 移除危险默认值。
- 启动时强制要求 `ENCRYPTION_MASTER_KEY` 与 `ENCRYPTION_SALT`。
- 为历史环境提供迁移说明与轮换策略。

## P1 修复项

### 9. 模型选择未完全按用户选择生效

- 严重级别：P1
- 位置：`backend/app/services/openai_service.py:107-108`
- 关联代码：`frontend/src/pages/PromptEditor.vue:1061-1068,1149-1156`
- 关联代码：`frontend/src/pages/CompareTest.vue:509-516`
- 问题说明：前端传入了 `model`，但实际调用优先使用 `AIConfig` 中保存的模型。
- 风险：
- 单次运行的模型切换不可信。
- 多模型对比结果可能并不是真正按所选模型产生。
- 修复建议：
- 明确“配置”和“模型”两个层次。
- 调用时优先使用用户本次选择的模型，配置只负责提供 `base_url` 和 `api_key`。
- 后端响应中回显最终真实使用的配置与模型。

### 10. 团队共享 Prompt 权限链路不一致

- 严重级别：P1
- 位置：`backend/app/api/run.py:70-79`
- 位置：`backend/app/api/batch_test.py:45-53`
- 位置：`backend/app/api/abtest.py:40-49`
- 对照：`backend/app/api/prompt.py:231-275,313-340`
- 问题说明：团队共享 Prompt 在详情与编辑接口中支持团队权限，但运行、批量测试、A/B 测试没有复用同样的权限判断。
- 风险：团队协作链路断裂，影响主功能闭环。
- 修复建议：
- 抽取统一的 Prompt 访问/编辑权限判定函数。
- 单次运行、批测、A/B、评论都统一接入。

### 11. A/B 测试页面结果流不完整

- 严重级别：P1
- 位置：`frontend/src/pages/CompareTest.vue:509-525,544-549`
- 问题说明：
- 调用测试接口后没有把结果写入 `testResults`。
- 加载历史记录时也没有把 `response.data.items` 赋给 `testHistory`。
- 风险：页面测试能力看起来存在，但核心交互链路不闭环。
- 修复建议：
- 测试创建成功后立即把返回值写入 `testResults`。
- 历史记录加载函数正确赋值列表。
- 为“当前结果”和“历史结果”增加统一数据适配层。

### 12. Prompt 复制功能实现不可靠

- 严重级别：P1
- 位置：`frontend/src/pages/PromptList.vue:264-281`
- 对照：`backend/app/api/prompt.py:191-208`
- 问题说明：复制功能使用列表项的 `prompt.content`，但列表接口并未返回 `content`。
- 风险：复制出来的 Prompt 可能为空内容或行为异常。
- 修复建议：
- 复制前先拉取详情接口。
- 或让后端提供专门的复制接口。

### 13. 多处列表页总数统计效率低

- 严重级别：P1
- 位置：`backend/app/api/execution_history.py:116`
- 位置：`backend/app/api/batch_test.py:260`
- 位置：`backend/app/api/admin.py:44`
- 位置：`backend/app/api/abtest.py:380`
- 问题说明：通过 `len(db.exec(statement).all())` 获取总数，会先全量查出记录再计数。
- 风险：数据量增大后列表性能明显下降。
- 修复建议：
- 统一改为 `select(func.count()).select_from(statement.subquery())`。
- 对管理后台和历史类查询优先修。

### 14. 系统配置测试接口存在明显异常分支问题

- 严重级别：P1
- 位置：`backend/app/api/system_config.py:252-260`
- 问题说明：异常分支中直接使用 `error_msg`，但这里没有先赋值。
- 风险：原始测试失败后，异常处理自身还可能再次报错。
- 修复建议：
- 先统一提取 `error_msg = str(e)`。
- 为网络异常、认证异常、模型异常分别输出明确消息。

## P2 修复项

### 15. 高价值敏感数据存放在 localStorage

- 严重级别：P2
- 位置：`frontend/src/utils/token.ts:6-28`
- 位置：`frontend/src/store/config.ts:115-119,233-245`
- 问题说明：token、用户信息、AI 配置等高价值数据落在 `localStorage`，后端失败时还会从本地回灌旧配置。
- 风险：一旦发生 XSS，凭据直接可被窃取。
- 修复建议：
- 优先移除 AI 配置本地回灌逻辑。
- token 至少评估迁移到更安全的会话方案。
- 降低前端持久化敏感信息的范围。

### 16. 新建 Prompt 草稿未按用户隔离

- 严重级别：P2
- 位置：`frontend/src/pages/PromptEditor.vue:842-901,949-966`
- 问题说明：草稿统一存放在固定键 `prompt_draft` 下。
- 风险：同一浏览器切换账号后，可能看到上一位用户的未发布草稿。
- 修复建议：
- 草稿键改为按用户 ID 隔离。
- 草稿结构中加入版本与来源信息。

### 17. 团队邀请链接默认自动加入

- 严重级别：P2
- 位置：`frontend/src/pages/Teams.vue:723-735`
- 问题说明：页面读取到 `?join=` 参数后会直接调用加入接口，没有二次确认。
- 风险：弱交互安全设计，用户误点链接即触发状态变更。
- 修复建议：
- 改为弹出确认框，展示团队名称、角色、有效期后再提交。

### 18. 回复输入框的 @提及 逻辑异常

- 严重级别：P2
- 位置：`frontend/src/components/PromptComments.vue:223-229,303-309,525-535`
- 问题说明：回复场景复用了提及逻辑，但选中提及用户时写回的是新评论输入框而不是回复输入框。
- 风险：回复提及体验异常，功能不完整。
- 修复建议：
- 区分“当前活跃输入框”的上下文。
- 将提及逻辑从单一状态改为可复用 composable 或局部状态机。

### 19. 缺少数据库迁移体系

- 严重级别：P2
- 位置：`backend/app/core/database.py:23-25`
- 关联代码：`backend/app/main.py:75`
- 问题说明：当前仍依赖 `SQLModel.metadata.create_all()` 自动建表，没有迁移体系。
- 风险：后续表结构演进、字段改名、历史数据迁移都会很危险。
- 修复建议：
- 引入 Alembic。
- 停止在应用启动时自动建表。
- 建立迁移脚本与发布流程。

### 20. 缺少自动化测试与构建校验

- 严重级别：P2
- 现状：仓库中未发现前后端测试目录。
- 额外发现：当前环境执行 `frontend` 的 `npm run build` 失败，提示缺少 `vue-tsc` 命令，说明本机依赖未完整安装或环境未就绪。
- 风险：回归问题难以控制，重构成本高。
- 修复建议：
- 前端至少补充关键页面的组件/交互测试。
- 后端至少补充认证、权限、Prompt CRUD、运行链路的 API 测试。
- CI 中加入前端构建、后端静态检查与测试。

## 建议的修复顺序

### 第一阶段：安全收口

目标：先把高风险面关掉。

建议顺序：
- 修复 JWT 密钥强制校验
- 修复评论接口权限判断
- 去除评论与 AI 输出的直接不安全 HTML 渲染
- 修复 `is_active` 校验缺失
- 重构 AI 密钥回传与测试连接链路
- 停止用户级 `api_key` 明文暴露

### 第二阶段：核心流程打通

目标：让主功能真的闭环可用。

建议顺序：
- 修复模型选择真实生效
- 统一团队共享 Prompt 权限链路
- 修复 A/B 测试结果展示与历史列表加载
- 修复 Prompt 复制功能
- 修复系统配置测试接口异常分支

### 第三阶段：工程化增强

目标：降低后续演进成本。

建议顺序：
- 替换低效统计查询
- 引入数据库迁移体系
- 建立测试与 CI
- 规范日志与错误处理
- 收缩前端本地敏感数据持久化

## 后续可拓展功能

以下方向都值得做，但建议在 P0/P1 问题收口后推进。

## 一、模型路由与 AI 能力平台化

### 目标

把当前“配置一个模型然后调用”的能力升级为真正的平台能力。

### 建议功能

1. 配置选择与模型选择解耦
2. 同一配置下支持多个模型切换
3. 模型 fallback 机制
4. 按成本、速度、质量做路由策略
5. 每次调用记录真实命中的 provider / model / base_url
6. 用户级、团队级默认模型策略

### 价值

- 让多模型对比真正可信
- 让企业场景更容易接入多家供应商
- 为后续成本优化和 SLA 做基础设施准备

## 二、实验平台深化

### 目标

把当前的“测试功能”升级为可复现、可比较、可归档的实验平台。

### 建议功能

1. 测试数据集管理
2. 测试数据集版本化
3. 基线结果与回归检测
4. 批量测试报告导出
5. A/B 测试结果可视化增强
6. 历史实验对比
7. 评分维度可配置
8. 允许人工评分和 AI 评分混合

### 价值

- 更适合团队持续优化 Prompt
- 能沉淀实验结果，而不是一次性使用

## 三、团队协作闭环

### 目标

把团队功能从“共享列表”升级为“共享工作流”。

### 建议功能

1. 团队共享 Prompt 的统一执行权限
2. 团队级评论、评审、结论归档
3. 团队模板库
4. 团队级实验与报表
5. 团队成员通知与提及提醒
6. 团队活动时间线
7. 团队空间内的收藏、推荐与热点 Prompt

### 价值

- 让协作不只停留在“能看到”
- 提升团队沉淀与复用效率

## 四、Prompt 知识库与搜索增强

### 目标

让 Prompt、模板、评论、评审、实验结果真正可检索、可复用。

### 建议功能

1. 全文搜索
2. 按标签、用途、团队、质量评分筛选
3. 相似 Prompt 推荐
4. 最佳实践标签体系
5. 从评论与评审中自动提取经验总结
6. 收藏夹、专题集、知识卡片

### 价值

- 平台会越来越像“Prompt IDE + 知识库”而不是简单表单系统

## 五、异步任务与可观测性

### 目标

把长耗时操作从同步请求改为可追踪任务。

### 建议功能

1. A/B 测试异步任务化
2. 批量测试异步任务化
3. AI 评测异步任务化
4. 报告生成异步任务化
5. 任务状态中心
6. 失败重试机制
7. 慢请求与失败分类统计
8. 成本告警、配额告警

### 价值

- 提高体验稳定性
- 支撑更大规模的实验任务

## 六、安全与合规增强

### 目标

让平台更适合长期使用和多人协作。

### 建议功能

1. 更细粒度 RBAC
2. 操作审计统一化
3. 敏感信息检测前置到执行链路
4. Prompt 共享前安全检查
5. API Key 轮换与失效机制
6. 登录风控与异常行为检测
7. 更安全的会话持久化方案

## 推荐实施路线

### 里程碑 A：稳定性与安全

交付目标：
- 关闭高风险安全面
- 修复核心主流程 bug
- 建立最小可回归测试集

### 里程碑 B：协作与实验闭环

交付目标：
- 团队共享 Prompt 可完整运行、评审、测试
- A/B 与批量测试形成稳定结果流
- 历史结果与报告可沉淀

### 里程碑 C：平台化升级

交付目标：
- 模型路由、实验资产、任务中心、知识库沉淀
- 平台具备持续扩展能力

## 最后建议

当前最不建议的做法：
- 继续快速叠加新页面
- 在未收口安全问题前继续加强前端富渲染
- 在没有迁移和测试体系的情况下持续改表结构

当前最建议的做法：
- 先完成 P0
- 再完成 P1 里和主流程直接相关的项
- 同步补最小测试集和数据库迁移体系
- 之后再进入大功能扩展阶段

## 建议下一步任务拆分

1. 安全修复专项
2. AI 配置与模型选择链路修复专项
3. 团队共享权限统一专项
4. A/B 与批量测试结果流修复专项
5. 工程化专项：测试、迁移、日志、构建
